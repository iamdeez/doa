## e2e 자동화 부재 (상세)

**현상**: 007 은 관리자 콘솔 UI 화면 6종을 추가하나(통계·정산·사용자·감사 로그·판매자 승인·배너), 화면 흐름
(통계 렌더 / 정산 표 / 사용자 무한 스크롤 / 감사 로그 표 / 판매자 승인 / 배너 생성·토글·삭제)에 대한 e2e/단위
테스트가 없다. 검증은 정적 코드 리뷰 + console 빌드/타입체크로 갈음했다.

**근본 원인 (UI 화면 성격 + 검증 갈음)**:
- 본 차수는 화면 추가 + 판매자 승인 실데이터화로, 라우트 컴파일·타입 안전성을 console typecheck/build 로
  확인했다. 사용자 상호작용 흐름(판매자 승인→invalidate, 배너 생성 검증→제출→invalidate, 삭제→invalidate,
  사용자 더 보기)은 e2e 로 회귀 방지하는 것이 이상적이나 본 차수에 포함하지 않았다(004·005·006 과 동일 패턴).

**위험도**: 낮음. 화면 로직이 명료(useQuery·useInfiniteQuery·useMutation·상태 분기)하고 console 빌드/타입체크가
라우트 컴파일·타입 회귀를 포착한다. 다만 상호작용 흐름의 미묘한 회귀(invalidate 누락·처리 중 행 식별 경로)는
e2e 없이는 탐지가 어렵다.

**권장 수정 방향**: 후속에 Playwright 등으로 (1) 플랫폼 통계 카드 렌더, (2) 전체 정산 표·상태 Badge, (3) 사용자
무한 스크롤(더 보기), (4) 감사 로그 표, (5) 판매자 승인→대기 목록 갱신, (6) 배너 생성→목록 갱신·활성 토글·삭제
e2e 를 추가한다(GAP-007-01).

## 관리자 화면 후속 미구현 (상세)

**현상**: 관리자 화면에 네 가지 후속 항목이 미구현이다.
- (1) **배너 삭제 확인 다이얼로그**: 배너 삭제는 `danger` 버튼 클릭 시 `deleteBanner` 를 **즉시** 호출하며,
  파괴적 조치 재확인(`AlertDialog`)이 없다. 실수 삭제 위험이 있으나 배너는 재생성 가능하다.
- (2) **클라이언트 권한 노출 차단**: `layout.tsx` 의 `visible` 필터는 seller 섹션만 `isSeller` 로 가리고 admin
  섹션은 항상 노출한다. 비관리자에게도 admin 메뉴(배너·전체 정산·플랫폼 통계·사용자·감사 로그·판매자 승인)가
  보인다. 데이터 보호는 백엔드 `AdminGuard`(403)가 강제하나, UI 노출은 UX 결함이다.
- (3) **배너 편집(필드 수정)**: 화면은 활성 토글(`updateBanner({ isActive })`) 외 제목·이미지·위치 등 필드
  편집 UI 를 제공하지 않는다. `updateBanner` facade 는 `UpdateBannerRequest`(부분 갱신)를 지원한다.
- (4) **낙관적 업데이트**: approveSeller·createBanner·updateBanner·deleteBanner mutation 은 서버 응답 후
  invalidate 방식(낙관적 미적용)이다.

**근본 원인 (Phase 3 최소 구현 — 범위 분리)**:
- Phase 3 는 관리자 운영 화면 제공이 목표이며, 파괴적 조치 재확인·UI 권한 필터·배너 편집·낙관적 UX 는 후속으로
  분리했다(spec §범위 외). 현재 구현도 처리 중 비활성화·에러 표시·invalidate 로 기본 안전성·정합성을 확보한다.

**위험도**: 낮음(의도된 범위 분리). 권한은 백엔드 `AdminGuard` 가 최종 강제하며(클라이언트 노출은 데이터 보호가
아님), 배너 삭제는 재생성 가능하고 모든 mutation 이 목록을 invalidate 하여 정합성을 유지한다.

**권장 수정 방향**: 후속에서 (1) 배너 삭제 `AlertDialog` 재확인, (2) `layout.tsx` 에 `isAdmin` 기반 admin 섹션
필터, (3) 배너 편집 다이얼로그, (4) `onMutate` 낙관적 업데이트 + 롤백을 적용하고 각각의 검증 테스트를 추가한다
(GAP-007-01).

## 응답 view 타입 한시 (상세)

**현상**: 관리자 통계·정산·사용자·감사·판매자·배너 응답은 백엔드가 Prisma 엔티티를 반환하고 OpenAPI 응답
content 가 미주석이다(001 GAP-001-01 연속). 따라서 003 타입드 client 대신 `@doa/shared-types` 전이형 view
타입(금전 string) 9종을 한시 정의했다(004·006 와 동일 패턴). 정산은 006 `SettlementView` 를 재사용했다.

**근본 원인 (백엔드 응답 스키마 미정의 — 점진 보강)**:
- 백엔드 컨트롤러가 응답 DTO + `@ApiResponse({ type })` 를 부여하지 않아 응답이 타입 미주석이다. 프론트는
  생성 타입에서 응답 타입을 얻지 못해 수기 view 타입으로 한시 표현한다.

**위험도**: 낮음(점진 보강 대상). view 타입은 공유 패키지(`shared-types`)에 단일 정의되어, 백엔드 응답 DTO
보강 후 한 곳에서 생성 타입(`Schemas['...']`)으로 교체 가능하다.

**권장 수정 방향**: 백엔드에 관리자 도메인별 응답 DTO + `@ApiResponse({ type })` 를 보강하고 코드젠 재생성
(`openapi:gen` → `gen`)하면 view 타입을 생성 타입으로 대체할 수 있다(GAP-007-01 / 006 GAP-006-01 / 004
GAP-004-01 / 001 GAP-001-01 연속). 금전 필드(`PlatformOverview.totalSales`)는 Decimal→문자열이므로 대체
후에도 `string` 유지를 확인한다(P-005).
