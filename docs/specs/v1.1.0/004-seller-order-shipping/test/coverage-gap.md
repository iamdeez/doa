## e2e 자동화 부재 (상세)

**현상**: 004 는 판매자 주문·배송 UI 화면 2개를 추가하나, 화면 흐름(목록 렌더 → 주문 확인 → 송장 등록 →
배송 전이 → 추적 조회)에 대한 e2e/단위 테스트가 없다. 검증은 정적 코드 리뷰 + console 빌드/타입체크로
갈음했다.

**근본 원인 (UI 화면 성격 + 검증 갈음)**:
- 본 차수는 화면 추가로, 라우트 컴파일·타입 안전성을 console typecheck/build 로 확인했다. 사용자 상호작용
  흐름(클릭→mutation→상태 전이→리렌더)은 e2e 로 회귀 방지하는 것이 이상적이나 본 차수에 포함하지 않았다.

**위험도**: 낮음. 화면 로직이 명료(상태별 switch 분기·mutation·useState)하고 console 빌드/타입체크가 라우트
컴파일·타입 회귀를 포착한다. 다만 상호작용 흐름의 미묘한 회귀(예: invalidate 누락·세션 state 경로)는 e2e
없이는 탐지가 어렵다.

**권장 수정 방향**: 후속에 Playwright 등으로 (1) 판매자 목록 렌더·상태별 조치 표시, (2) confirm → 목록
갱신, (3) 송장 등록 → 발송 상태 표시, (4) 배송중·배송완료 전이 → 버튼 비활성화, (5) 추적 이력 렌더 e2e 를
추가한다(GAP-004-01).

## BE-GAP 의존 시나리오 미구현 (상세)

**현상**: 두 시나리오가 백엔드 엔드포인트 부재로 미구현이다.
- (1) **ship 재진입 기존 송장 복구**: 주문→송장 조회 엔드포인트(`GET /shipments?orderId` 또는 주문 응답에
  shipment 포함)가 부재하여, 이미 발송된 주문에 ship 페이지로 재진입하면 기존 shipment id 를 복구할 수 없다.
  현재는 송장 등록 직후 세션 state 의 shipment 로 상태변경·추적이 동작한다(세션 내 완결). 재진입 시 재등록
  시도는 백엔드가 400(주문이 preparing 아님)으로 거부한다.
- (2) **판매자 주문 상세(items)**: 판매자용 단건 주문 상세 엔드포인트가 부재하다(`GET /orders/:id` 는 구매자
  스코프). 따라서 ship 페이지는 주문 상세를 직접 가져오지 못하고 `useParams` 의 orderId 만 사용한다.

**근본 원인 (백엔드 계약 공백 — BE-GAP)**:
- 004 는 기존 백엔드 라우트(주문·배송 5개)를 소비하는 화면이며, 위 두 흐름에 필요한 라우트가 백엔드에 아직
  없다. 프론트 단독으로 해소할 수 없는 백엔드 의존 공백이다.

**위험도**: 중간(재진입 복구) ~ 낮음(주문 상세). 재진입 복구 부재는 송장 등록 후 페이지를 떠난 판매자가
세션 밖에서 배송 상태를 이어 관리할 수 없다는 UX 제약을 낳는다. 주문 상세 부재는 ship 페이지에 주문 품목
표시가 없다는 정보 제약이다.

**권장 수정 방향**: 백엔드에 (1) 주문→송장 조회(`GET /shipments?orderId` 또는 주문 응답에 shipment 포함),
(2) 판매자 스코프 단건 주문 상세(`GET /seller/orders/:id`, items 포함) 엔드포인트를 추가한다. 추가 후 ship
페이지가 진입 시 기존 송장을 복구하고 주문 상세를 표시하도록 보강한다(gaps.md GAP-004-01).

## rhf 검증·낙관적 업데이트 미적용 (상세)

**현상**: 송장 등록 폼은 react-hook-form + zod 스키마 검증 없이 제어 컴포넌트(`useState`) + 빈 값
비활성화(`!carrier || !trackingNumber`) 수준이다. confirm·updateStatus mutation 은 낙관적 업데이트 없이
서버 응답 후 `setState`/`invalidateQueries` 한다.

**근본 원인 (Phase 1 최소 구현 — 범위 분리)**:
- Phase 1 은 판매자 주문 이행 흐름 제공이 목표이며, 폼 검증 정교화·낙관적 UX 는 Phase 2 로 분리했다(spec.md
  범위 외). 현재 구현도 빈 값 비활성화·처리 중 비활성화·에러 표시로 기본 안전성은 확보한다.

**위험도**: 낮음(의도된 범위 분리). 입력 검증은 백엔드 DTO(class-validator)가 강제하며, mutation 은 서버
응답 기준으로 상태를 갱신하여 정합성을 유지한다.

**권장 수정 방향**: Phase 2 에서 (1) 송장 등록 폼을 rhf+zod 스키마 검증으로 전환(택배사·운송장 형식 검증),
(2) confirm·updateStatus 에 `onMutate` 낙관적 업데이트 + 실패 롤백을 적용하고 각각의 검증 테스트를 추가한다
(GAP-004-01).
