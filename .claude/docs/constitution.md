# Project Constitution

> 이 문서는 프로젝트의 **불변 원칙**을 정의한다.
> 모든 설계·구현·검증 작업 전에 반드시 읽는다.
> constitution 조항이 전역 규칙과 충돌하는 경우, constitution이 우선한다.
>
> - **변경 원칙**: 조항은 팀 전체의 합의 없이 변경하지 않는다.
> - **기준 커밋**: 문서에 commit hash·변경 이력을 기록하지 않는다. 기준 커밋이 필요하면 파이프라인 baseCommit (`pipeline-log.md` "단계 시작" 이벤트) 또는 `git log -- <파일>` 을 사용한다.

---

## 1. 프로젝트 원칙

### P-001. 모듈 경계 원칙

각 도메인 모듈은 자신의 DB 스키마 테이블에만 직접 접근한다. 타 도메인 데이터는 공개 서비스 인터페이스 또는 인-프로세스 도메인 이벤트로만 획득한다.

- 모듈 간 직접 DB 교차 쿼리 금지 (다른 스키마 테이블 직접 JOIN 포함).
- 모듈 간 통신은 NestJS DI를 통한 서비스 메서드 호출 또는 EventEmitter 이벤트만 허용.
- 각 모듈은 `controller` · `service` · `repository` · `events` 4계층 구조를 준수한다.

**Gates 질문**: 이 구현이 다른 도메인 모듈의 스키마 테이블을 직접 참조하거나 쿼리하지 않는가?

---

### P-002. AWS 의존 금지 원칙

AWS 전용 SDK, 서비스, 인프라 구성을 신규 코드에 도입하지 않는다.

- `@aws-sdk/*`, `aws-xray-sdk`, `winston-cloudwatch`, `aws4` 등 AWS 전용 패키지 신규 추가 금지.
- Cognito, EventBridge, SQS, DynamoDB, ElastiCache, CloudWatch, Lambda를 신규 의존으로 추가 금지.
- 파일 스토리지는 S3 호환 인터페이스(`@aws-sdk/client-s3` 내 S3 호환 엔드포인트)만 허용하며, Cloudflare R2를 기본으로 한다.

**Gates 질문**: 이 구현이 AWS 전용 서비스 또는 SDK에 새로운 의존을 추가하지 않는가?

---

### P-003. 단일 DB 원칙

관계형 데이터·캐시·큐·세션·NoSQL 용도를 모두 단일 PostgreSQL 인스턴스(스키마 분리)로 처리한다. 외부 데이터 저장소 추가는 충분한 근거가 있어야 한다.

- 신규 외부 데이터 저장소(별도 DB, 별도 Redis, 별도 메시지 브로커)를 도입할 때는 spec.md에 근거를 명시하고 사용자 승인 후 진행한다.
- 비동기 잡·큐는 pg-boss(PostgreSQL 기반)를 기본으로 한다.
- 인-앱 LRU 캐시로 충분한 경우 외부 캐시(Upstash 등)를 도입하지 않는다.

**Gates 질문**: 이 구현이 단일 PostgreSQL 외 새로운 외부 데이터 저장소를 추가하지 않는가? (추가 시 spec.md 근거·승인 확인)

---

### P-004. 클라우드 중립 원칙

배포 플랫폼(Fly.io)에 종속되는 API·SDK·설정을 최소화하여 다른 플랫폼으로 이전 가능하게 한다.

- 표준 Docker 컨테이너, 표준 PostgreSQL, S3 호환 스토리지 인터페이스만 사용한다.
- Fly.io 전용 SDK나 플랫폼 전용 API에 비즈니스 로직을 결합하지 않는다. 인프라 설정(`fly.toml`, `flyctl`)은 배포 레이어에만 위치한다.

**Gates 질문**: 이 구현이 Fly.io 전용 API에 비즈니스 로직을 결합하지 않는가?

---

### P-005. 결제·정산 정합성 원칙

결제·환불·정산 흐름에서 금전 관련 상태 변경은 반드시 단일 트랜잭션 + outbox 패턴으로 처리한다.

- `payment`, `settlement` 모듈의 상태 변경과 후속 도메인 이벤트 발행은 동일 트랜잭션 내 outbox 기록으로 보장한다.
- 결제·환불 API는 멱등성 키(idempotency key)를 필수로 적용한다.
- 금전 관련 수치 연산은 정수(최소 단위: 원) 또는 Prisma Decimal 타입만 사용하고 부동소수점 연산을 금지한다.

**Gates 질문**: 이 구현이 결제·환불·정산 상태 변경을 outbox 패턴과 멱등성 키 없이 처리하지 않는가?

---

### P-006. 테스트 원칙

모든 기능 요구사항(FR-XXX)은 수용 기준(SC-XXX)과 대응하는 검증 시나리오를 갖춰야 한다.

- SC-XXX 없는 FR-XXX의 구현은 시작하지 않는다.
- 테스트 커버리지를 낮추는 변경은 허용하지 않는다.

**Gates 질문**: 모든 FR-XXX에 대한 검증 시나리오가 spec.md에 정의되어 있는가?

---

### P-007. 스펙 범위 원칙

구현은 spec.md에 명시된 범위를 벗어나지 않는다.

- spec 외 리팩토링, 기능 추가, 성능 개선은 별도 spec으로 분리한다.
- 기존 18개 도메인 핵심 기능 이전이 우선이며, 신규 비즈니스 기능 추가는 재구축 완료 후 별도 spec으로 진행한다.

**Gates 질문**: 이 구현이 spec.md 범위를 벗어나는 변경을 포함하지 않는가?

---

## 2. 예외 처리 방식

constitution 위반이 불가피한 경우 아래 절차를 따른다.

1. `plan.md`의 **"예외 사항"** 항목에 위반 조항과 근거를 명시한다.
2. 대안(위반 범위 최소화, 단계적 해소 계획 등)을 함께 기재한다.
3. 사용자의 확인을 받은 후 구현을 진행한다.
4. 구조적 위반이라면 spec 범위를 재조정하거나, 팀 합의 후 constitution 조항 자체를 개정한다.
